# MinerU Paper Reading Skill

This skill is for reading research papers from **MinerU output folders**.

The user usually has a local folder produced by MinerU, not only a single PDF. The folder may contain:

- original `.pdf`
- layout `.pdf`
- one or more `.md` files
- `images/`
- page screenshots
- figure/table/formula images
- `.json` files containing layout, OCR, page spans, formulas, tables, or metadata
- table files such as `.html`, `.csv`, `.xlsx`
- intermediate files generated by MinerU

## 0. Design goal

The user wants two workflows.

### Workflow A: 预精读决策报告

The user may say something like:

> 为我先精读一下这里的文章，然后我会根据你的输出而选择要不要亲自精读。

This does **not** mean full sentence-by-sentence close reading.

It means:

1. read the paper according to Keshav's method;
2. understand the paper enough to explain it accurately;
3. answer:
   - 为什么写这篇论文？
   - 这篇论文写了什么？
   - 每章每节分别讲什么？
   - 关键图表如何支撑对应章节？
   - 文章总结是什么？
   - 用户后续是否应该亲自精读？

The answer must be detailed enough for the user to decide whether to continue reading personally.

### Workflow B: 逐句精读

The user may say:

> 我决定精读这篇论文。请逐句精读。

Then you must read sentence by sentence:

- one original sentence;
- one Chinese translation;
- one explanation;
- relevant figure/table/formula/reference integrated into the explanation;
- section summary after each section;
- chapter interpretation after each chapter;
- full-paper interpretation after the whole paper.

## 1. Trigger conditions

Use this skill when the user says any of the following:

- "为我先精读一下这里的文章，然后我会根据你的输出而选择要不要亲自精读"
- "我会根据你的输出选择要不要亲自精读"
- "帮我先精读一下这篇论文，看看我后续要不要自己精读"
- "判断这篇论文值不值得后续精读"
- "我想了解这篇论文的总体过程和发展历程"
- "这篇论文为什么写、写了什么、总结是什么"
- "帮我阅读 MinerU 输出"
- "读取 mineru_result / hybrid_auto / Markdown + images"
- "按 How to Read a Paper 的方法读论文"
- "逐句精读"
- "一句原文，一句翻译，一句解析"
- "图文结合讲解论文"
- "不要把文字和图表分开"
- "每节解析、每章解析、全文解析"

Do not use this skill for unrelated coding tasks.

## 2. User interest focus

The user's prompt may include a specific interest, for example:

- "我想了解总体的 post-training 过程和发展历程"
- "我关注 GRPO / RLHF / DPO / DeepSeek-R1"
- "我关注数据集"
- "我关注 open problems"
- "我关注多模态/领域适配/效率"

When a focus is provided:

1. still read the whole paper;
2. make the whole-paper report complete;
3. allocate more explanation depth to the user's focus;
4. explicitly connect each relevant chapter/subsection to that focus;
5. do not silently skip unrelated chapters, but mark them as lower priority if appropriate.

Example:

If the user says they care about "overall post-training process and development history", emphasize:

- abstract/introduction motivation;
- historical timeline;
- taxonomy;
- the sequence from SFT to RLHF/DPO to reasoning RL to efficiency and adaptation;
- figures that explain development paths;
- conclusion and open problems.

## 3. Mandatory source handling: Markdown first, PDF as auxiliary verification

Always handle MinerU output in this order:

1. **Find the MinerU output folder.**
   Common names include:
   - `mineru_result`
   - `output`
   - `outputs`
   - `hybrid_auto`
   - a folder named after the paper title

2. **Identify the main Markdown.**
   Usually the largest `.md` file or the `.md` matching the PDF title.

3. **Identify the original PDF and layout PDF.**
   Prefer the original PDF as the authoritative backup.
   Use layout PDF/page images when useful for visual positioning.

4. **Read Markdown as the main spine.**
   Use it to locate:
   - title
   - abstract
   - introduction
   - table of contents
   - section/subsection headings
   - figures/tables/formulas/captions
   - conclusion
   - references

5. **Use images/tables/formulas/JSON as linked assets.**
   Whenever the Markdown references a figure/table/formula, inspect the linked asset when possible.

6. **Use the original PDF as an auxiliary check after reading Markdown.**
   Use PDF to verify:
   - title/abstract/conclusion;
   - important figures/tables/formulas;
   - suspicious OCR or Markdown parsing issues;
   - any sentence where Markdown quality affects interpretation.

Do not claim "MinerU Markdown has encoding/OCR errors" unless you provide exact examples:
- file path;
- line number or nearby heading;
- original Markdown fragment;
- suspected issue;
- PDF check result;
- impact on reading.

## 4. Internal source inventory rule

Before giving a paper-reading answer, internally inspect the source inventory.

PowerShell examples:

```powershell
Get-ChildItem -Recurse -Filter *.md
Get-ChildItem -Recurse -Filter *.pdf
Get-ChildItem -Recurse -Filter *.json
Get-ChildItem -Recurse -Include *.png,*.jpg,*.jpeg,*.webp
```

Bash examples:

```bash
find . -name "*.md"
find . -name "*.pdf"
find . -name "*.json"
find . \( -name "*.png" -o -name "*.jpg" -o -name "*.jpeg" -o -name "*.webp" \)
```

Internally identify:

- Main Markdown
- Original PDF
- Layout PDF
- Images folder
- JSON/content/layout files
- Table/formula assets
- PDF auxiliary check status

### Do not display source inventory by default

For Workflow A, do **not** output a visible `Source inventory` section unless the user explicitly asks for diagnostic information.

Instead, briefly mention source evidence naturally inside the answer when needed, for example:

> 我以主 Markdown 为阅读主线，并用原始 PDF/关键图片辅助核验了标题、摘要、目录、关键图表和结论。

If the user asks for troubleshooting, diagnostics, or verification, then show the source inventory.

## 5. Keshav reading framework

The skill follows S. Keshav's paper-reading method.

### 5.1 First pass

Read:

1. title;
2. abstract;
3. introduction;
4. section and subsection headings;
5. conclusion;
6. references/bibliography.

Internally answer the five Cs:

| C | Required question |
|---|---|
| Category | What type of paper is this? Survey, method paper, systems paper, measurement paper, dataset paper, benchmark paper, theory paper, etc. |
| Context | What background problem does it address? Which research area and prior work does it connect to? |
| Correctness | Are the assumptions, framing, evidence, and organization plausible? For a survey, does the taxonomy look reasonable? |
| Contributions | What does the paper claim to contribute? Are those contributions real or mostly organizational? |
| Clarity | Is the writing/structure clear? Are figures/tables useful? Are there parsing or PDF quality issues? |

### Do not display the five Cs by default

The five Cs are an internal reading method.

For Workflow A, do **not** output a visible "Five Cs" table unless the user explicitly asks.

Use the five Cs to improve the report, especially:

- paper category;
- research context;
- contribution quality;
- structure clarity;
- whether the paper is worth further reading.

### 5.2 Second pass

Read more carefully:

- main content of each section;
- subsection structure;
- key definitions and claims;
- key figures/tables/formulas;
- conclusion/open problems;
- references only as background signals, unless the user asks for reference analysis.

Pay special attention to figures, diagrams, graphs, tables, and formula blocks. Do not explain them separately at the end only. They must be explained inside the section where they appear.

### 5.3 Third-pass light check for Workflow A

For Workflow A, do not perform full reviewer-level third pass or full sentence-by-sentence reading.

However, do a light third-pass check:

- Can the paper's overall logic be reconstructed?
- What is the core argument chain?
- Where are the strongest and weakest parts?
- Which chapters deserve real deep reading?
- Which parts are only maps/indexes and should not be treated as final technical authority?

For Workflow B, use the full intensive reading process.

## 6. Main Workflow A: 预精读决策报告

Use this mode when the user says:

> 为我先精读一下这里的文章，然后我会根据你的输出而选择要不要亲自精读。

This mode means:

- Do **not** do full sentence-by-sentence close reading yet.
- Do **not** answer with only "值得读/不值得读".
- Apply Keshav first pass + second pass + light third-pass check.
- Produce a detailed paper-level, chapter-level, and subsection-level explanation.
- Let the user decide whether to continue to intensive reading.

## 6.1 Output these parts only

Use Chinese by default.

Do not output `Source inventory`, `Five Cs`, or `关键参考文献线索` by default.

Use this structure:

```markdown
# 预精读阅读报告

## 1. 为什么写这篇论文？
从 Abstract 和 Introduction 中回答。必须说明：
- 这篇论文面对什么研究背景？
- 作者认为已有研究/已有综述缺什么？
- 为什么这个问题现在重要？
- 作者写这篇论文想解决什么“组织知识/梳理领域/提出框架”的问题？
- 这篇论文和用户关注点有什么关系？

## 2. 这篇论文写了什么？
这一部分必须详细，允许较长。不要只给几条短 bullet。

### 2.1 全文主线
用多段文字说明整篇论文的中心脉络。
如果用户提供了关注点，必须把全文主线和用户关注点连接起来。

### 2.2 核心贡献详解
这部分要写长、写细。
必须区分：
- 作者声称的贡献；
- 实际上对读者有价值的贡献；
- 可能存在的过度宣称或局限；
- 这些贡献对用户关注点意味着什么。

### 2.3 章节级概括
先给总表：

| 章节 | 这一章写什么 | 为什么放在这里 | 关键图表/公式 | 是否值得后续精读 |
|---|---|---|---|---|

然后对每一章写详细解释。

### 2.4 小节级概括
对每一章下的小节进行概括。

格式：

#### 第 X 章：章节名
- X.1 小节名：约 200 字左右。说明这一小节讲什么、为什么重要、和全文主线的关系、是否值得后续精读。
- X.2 小节名：约 200 字左右。
- X.3 小节名：约 200 字左右。

要求：
- 不要跳过可见小节。
- 不要只写一句话。
- 不要用空泛判断替代内容解释。
- 如果某小节价值较低，也要说明为什么只需略读。
- 如果小节和用户关注点强相关，要写得更详细。

## 3. 关键图表/公式/表格解释
重要：图表解释必须回到对应章节和对应文段中，不要把图表和文字割裂。

输出格式：

### 第 X 章/第 X.Y 节中的 Figure/Table/Equation
- 触发文本/章节：
- 图表/公式本身说明：
- 它如何支撑当前章节的论证：
- 它和用户关注点的关系：
- 是否需要用户后续亲自细看：

不要只列图名。必须解释图表在论文论证中的作用。

## 4. 文章总结
这一部分要写长、写完整。
必须说明：
- 论文最终想表达什么；
- 作者的整体结论是什么；
- 全文的知识结构是什么；
- 它对该领域的意义是什么；
- 它的局限在哪里；
- 它适合作为地图文献、背景文献、核心技术文献，还是引用索引；
- 用户如果继续读，应该带着什么问题去读。

## 5. 给用户的后续选择依据
给清晰建议：
- 必须后续精读的章节；
- 选择性精读的章节；
- 只需略读的章节；
- 可以跳过的章节；
- 如果用户目标是 A/B/C，应该怎么读。
```

## 6.2 Required quality for Workflow A

Workflow A should be substantial. It is not a short summary.

Minimum expectations for a long paper:

- "为什么写这篇论文" must be grounded in abstract and introduction.
- "这篇论文写了什么" must include every major chapter.
- "核心贡献详解" must be longer than a few bullets.
- "小节级概括" must include all visible subsections from the table of contents.
- Each visible subsection summary should aim for about 200 Chinese characters when the paper is long and the user asks for detail.
- Key figures/tables/formulas must be placed under their relevant chapters/sections.
- The conclusion must be detailed and connected back to the motivation.
- Reference scanning should be done internally, but do not output a reference-clue section unless the user explicitly asks.

## 7. Main Workflow B: Intensive sentence-by-sentence close reading

Use this mode only when the user says:

- "我已经决定精读"
- "开始逐句精读"
- "一句原文，一句翻译，一句解析"
- "对第 X 节逐句精读"
- "每一句都要"

### 7.1 Strict sentence-by-sentence rule

In Intensive Mode, read sentence by sentence.

For every sentence, output:

```markdown
### Sentence N

**原文：** ...
**翻译：** ...
**解析：** ...
```

The explanation must include when relevant:

- terminology;
- grammar or structure if needed;
- what role the sentence plays in the paragraph;
- whether it is motivation, definition, claim, evidence, transition, limitation, or conclusion;
- linked figure/table/formula/reference;
- PDF/Markdown evidence location.

### 7.2 Integrate figures/tables/formulas into the relevant sentence

Do not separate text reading and figure reading.

When a sentence mentions or depends on a figure/table/formula:

```markdown
**关联图表/公式：**
- Asset:
- Caption:
- What it shows:
- How it supports this sentence:
- What the user should notice:
```

This must appear inside the explanation of that sentence or immediately after that sentence, not in a disconnected section at the end.

### 7.3 Section and chapter summaries during intensive reading

After finishing every section:

```markdown
## 本节解析
- 本节核心问题：
- 本节讲了什么：
- 本节关键概念：
- 本节关键图表/公式：
- 本节和全文主线的关系：
- 你需要记住什么：
```

After finishing every chapter:

```markdown
# 本章解读
- 本章在全文中的作用：
- 本章每节之间的逻辑关系：
- 本章最重要的图表/公式/参考文献：
- 本章对后续阅读的影响：
- 本章是否值得反复读：
```

After finishing the whole paper:

```markdown
# 全文解析
- 全文主线：
- 核心贡献：
- 关键证据链：
- 重要图表/公式/参考文献：
- 局限：
- 你最终应该如何理解这篇论文：
```

### 7.4 Batch rule for long papers

Do not attempt to output the entire paper in one answer.

Use batches:

1. Abstract + Introduction
2. Background / Overview / Formula Foundations
3. Main method or taxonomy sections
4. Experiments/results or survey tables
5. Datasets / Applications
6. Open Problems + Conclusion + References

At the end of each batch, ask whether to continue.

## 8. Context-linked multimodal reading

This is a strict rule for both workflows.

Do not read all text first and then discuss figures/tables later as a disconnected section.

Whenever the Markdown references:

- Figure / Fig. / 图
- Table / 表
- Equation / Eq. / 公式
- Algorithm / 算法
- Diagram / 示意图
- Plot / 曲线图
- Chart / 图表
- Architecture / 架构图
- Pipeline / 流程图
- Appendix figure/table

immediately:

1. locate the referenced asset;
2. inspect the asset when possible;
3. read the caption together with surrounding text;
4. integrate the asset into the explanation of the relevant sentence/paragraph/section;
5. explain what role it plays:
   - motivation;
   - definition;
   - taxonomy;
   - evidence;
   - result;
   - ablation;
   - method flow;
   - limitation;
6. check whether the visual/table/formula evidence supports the textual claim;
7. use the original PDF as auxiliary check for central or unclear assets.

## 8.1 Mermaid warning

MinerU may generate Mermaid diagrams from figures.

Treat Mermaid as secondary and potentially unreliable.

Priority order:

1. original PDF figure/table/formula;
2. extracted image/table/formula asset;
3. Markdown caption/surrounding text;
4. Mermaid reconstruction.

Never rely on Mermaid if it conflicts with the original image/PDF.

## 9. Reference protocol

References must be scanned internally, but do not display a "关键参考文献线索" section in Workflow A unless the user explicitly asks for it.

For the target paper, internally:

1. locate the `References`, `Bibliography`, or `Related Work`;
2. extract important citation IDs/names;
3. connect important citations back to where they are used;
4. identify:
   - foundational papers;
   - method papers;
   - baseline/model papers;
   - survey/background papers;
   - dataset papers.

Do not automatically deep-read all referenced papers unless the user asks.

Say:

- "当前论文引用了 X 作为……"
- not "X 证明了……" unless X itself was read.

If the user explicitly asks for reference clues, then output them.

## 10. Evidence rules

When giving evidence, include evidence naturally and only where useful.

Do not clutter the report with a visible source inventory or diagnostic details unless requested.

When evidence is needed, include:

- local file path;
- heading/section;
- Markdown line number if available;
- PDF page if checked;
- image/table/formula asset path if used.

Always distinguish:

- "论文原文说";
- "Markdown 中显示";
- "PDF 对照确认";
- "图表资产显示";
- "我的推断是".

If content is not found:

```markdown
我没有在已检查的 MinerU 输出中找到直接证据。已检查：
- ...
最接近的证据是：
- ...
```

## 11. Writing style

Use Chinese by default.

The user wants detailed, teaching-oriented explanations.

Prefer these headings and phrases:

- 为什么写这篇论文
- 这篇论文写了什么
- 全文主线
- 核心贡献详解
- 章节级概括
- 小节级概括
- 关键图表解释
- 文章总结
- 原文
- 翻译
- 解析
- 本节解析
- 本章解读
- 全文解析
- 图文结合
- PDF 对照检查

Avoid vague judgments unless supported by evidence.

## 12. Final checklist

Before answering, verify internally:

- Did I find the main Markdown?
- Did I find the original PDF/layout PDF?
- Did I read Markdown first?
- Did I use PDF as auxiliary verification where important?
- Did I inspect linked figures/tables/formulas when relevant?
- Did I keep figures/tables/formulas inside their relevant chapters/paragraphs/sentences?
- Did I scan references internally?
- Did I apply Keshav first/second pass and light third-pass check for Workflow A?
- Did I avoid displaying Source inventory by default?
- Did I avoid displaying the Five Cs by default?
- Did I avoid displaying reference clues by default?
- Did I make chapter/subsection explanations sufficiently detailed?
- Did I do sentence-by-sentence reading for Workflow B?
- Did I include section/chapter/full-paper interpretation in Workflow B?
- Did I give exact examples for any claimed parsing errors?
